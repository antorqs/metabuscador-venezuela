# pacientesve Source Integration Plan

## Goal
Integrate `pacientesve` from public TSV export in a way that is fast, resilient, and easy to operate during emergency response.

Source URL:
`https://docs.google.com/spreadsheets/d/1CSrJaBeCSo_l_0eoXI7SocThHKlRDztRMJaic1Ts0Ww/export?format=tsv&gid=0`

---

## Constraints & Strategy

- TSV is large (~875KB), so fetching on every search is too expensive.
- Data changes over time, so committing static TSV into repo is not acceptable.
- Use **server-side cached index** with periodic refresh (TTL).
- Search requests query in-memory index, not remote TSV.
- Keep last good data if refresh fails (high availability).

---

## High-Level Design

### 1) New source adapter
Create a dedicated adapter for this source (e.g. `pacientesVe.ts`) following existing `SourceAdapter` contract:
- `key`, `displayName`, `sourceUrl`
- `search(query): Promise<PersonResult[]>`

### 2) In-memory cache layer (module-local singleton)
Store:
- `rawTsvHash` (optional)
- `records: PersonResult[]`
- `normalizedSearchText[]` (optional optimization)
- `lastFetchedAt`
- `lastSuccessfulFetchAt`
- `isRefreshing`
- `lastError`

### 3) Refresh policy
- `CACHE_TTL_MS` default: `600000` (10 min)
- On search:
  - if cache fresh: serve immediately
  - if stale and not refreshing: trigger refresh
  - if stale and refreshing: serve stale cache
- On cold start with empty cache:
  - attempt blocking fetch once
  - if fails: return source error cleanly

### 4) Stale-while-revalidate behavior
- Never drop last good cache due to transient failures.
- If refresh fails:
  - keep old cache
  - update `lastError`
  - retry on next stale check window

---

## Parsing & Mapping Plan

### 5) TSV ingestion
- Fetch TSV URL with `cache: "no-store"`.
- Parse lines by `\n`, fields by `\t`.
- First row = headers.
- Build row objects by header name.
- Normalize headers (trim, lower-case, optional accent folding).

### 6) Field mapping to `PersonResult`
Map using detected columns (to be validated once sample headers are seen):
- `name`: best candidate among known name fields
- `age`: age-like column if present
- `cedula`: ID/cedula column if present
- `location`: location/hospital/municipio style columns
- `contact`: phone/contact columns
- `status`: map source status text to `missing|found|hospitalized|unknown`
- `photoUrl`: optional URL field if present
- `lastUpdated`: update/date column if present
- `profileUrl`: sheet/source URL

### 7) Search index normalization
For each record:
- create normalized text blob from key fields (`name`, `cedula`, `location`, etc.)
- normalization:
  - trim
  - uppercase
  - remove diacritics
  - collapse spaces
- query match: `includes(normalizedQuery)`

## Operational Controls

### 8) Env configuration
Add:
- `PACIENTESVE_TSV_URL` (default to provided URL)
- `PACIENTESVE_CACHE_TTL_MS` (default `600000`)
- `PACIENTESVE_MAX_RESULTS` (optional per-source cap, default from global)

### 9) Source catalog entry
Add to `SOURCE_CATALOG` as enabled once validated.
Include note field support for temporary disable if needed.

### 10) Observability
Expose minimal diagnostics in source error text when needed:
- `cache_empty_fetch_failed`
- `tsv_parse_error`
- `stale_cache_served`
(keep user-facing messages concise; detailed logs internal only)

---

## Acceptance Criteria

- Source responds without fetching TSV on every request.
- First request after cold start works or fails gracefully.
- Subsequent searches are fast (served from cache).
- Data refreshes automatically after TTL.
- Refresh failures do not break existing cached results.
- Source remains compatible with existing grouped UI/cards.
- `npm run lint` and `npm run build` pass.

---

## Risks & Mitigations

- **Header variability in TSV**:
  - mitigate with flexible header alias mapping.
- **Large memory footprint**:
  - keep only mapped fields + optional normalized blobs.
- **Cold start latency**:
  - consider optional startup prefetch later.
- **Google throttling/outage**:
  - stale cache fallback avoids hard failure.
